# 声笔 VoiceDraw — 设计文档（前后端系统版）

> 纯语音控制的绘图工具。不用鼠标、不用键盘，说出指令即可创作。
> 仓库：https://github.com/heziamo/voicedraw ｜ 线上：https://117.50.181.92:8443/

---

## 1. 项目目标与约束

| 维度 | 要求 | 应对策略 |
|---|---|---|
| 交互方式 | 仅语音，禁用鼠标/键盘 | 全部操作走「识别 → 后端解析 → 执行 → 语音播报」闭环；帮助、暂停、保存全部语音可达 |
| 指令理解准确性 | 容错口音、谐音、口语化 | 谐音矫正表 + 同义词词典 + 礼貌语剥离 + 中文数字解析 + 默认值兜底 |
| 响应延迟 | 越低越好 | 后端纯本地规则解析，实测端到端 **4ms**（局域）/数十 ms（公网）；瓶颈只剩浏览器 ASR |
| 复杂指令 | 支持拆解与顺序执行 | 后端分句器把一句话切成多条原子指令，逐句解析执行 |
| 工程化 | Python 前后端 + Git + 部署 | FastAPI 后端 + 原生前端，GitHub 版本管理，systemd + TLS 部署到云服务器 |

**唯一的非语音操作**：首次进入需点击一次「开始语音创作」。浏览器安全策略硬性要求麦克风必须由用户手势开启，无法绕过；此后所有操作均为语音。

## 2. 系统架构

```
┌─────────────────────── 浏览器（瘦客户端） ───────────────────────┐
│  麦克风采集 → Web Speech API 识别(zh-CN) → 实时字幕                  │
│  Canvas 渲染（逻辑坐标 1200×750 等比缩放） ← 后端返回的场景            │
│  TTS 语音播报 + 指令日志 + 状态指示（Linear 设计系统）                 │
└───────────────────────────────┬─────────────────────────────────┘
                    final 识别文本 │ HTTPS POST /commands
                    场景 + 结果     │ JSON
┌───────────────────────────────▼─────────────────────────────────┐
│  FastAPI 后端（Python 3.8+）                                        │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │ NLU 流水线（纯标准库，<5ms）                                   │  │
│  │  归一化 → 分句 → 意图解析(21) → 指代消解 → 任务规划 → 执行器     │  │
│  └────────────────────────────────────────────────────────────┘  │
│  会话状态机（场景图 + 撤销/重做栈 + 选中集）                          │
│  持久化层（可插拔：SQLite / JSON 文件）+ 指令审计日志                 │
└───────────────────────────────┬─────────────────────────────────┘
                                 │
                   ┌─────────────▼─────────────┐
                   │ SQLite (本地) / JSON (服务器) │
                   │ 会话状态 + commands 审计日志   │
                   └───────────────────────────┘
```

### 2.1 前后端职责划分

| 层 | 归属 | 职责 |
|---|---|---|
| 语音采集与识别 | **前端** | 只有浏览器能访问麦克风；Web Speech API 做 ASR；中间结果上屏、最终结果发后端 |
| 渲染 | **前端** | Canvas 2D，按后端返回的场景图重绘；逻辑坐标系与窗口尺寸解耦 |
| TTS 播报 | **前端** | 浏览器 SpeechSynthesis；回声抑制（播报时挂起识别） |
| **NLU 解析** | **后端** | 词典 + 规则把中文转结构化操作；这是核心，放后端便于演进/复用/多端共享 |
| **状态管理** | **后端** | 场景图、撤销/重做、选中集；前端无状态，刷新可从服务器恢复 |
| **持久化与审计** | **后端** | 会话落库，每条指令记审计日志 |

**为什么 NLU 放后端而不是前端？** ① 模板版（纯前端）的解析逻辑无法被其他客户端（小程序/App/IoT）复用；放后端后任何端只要发文本即可。② 词典与规则迭代不必让用户刷新缓存。③ 状态在服务端，天然支持刷新恢复、跨设备续画、审计回放。代价是每条指令多一次网络往返——但解析本身 <5ms，相比 ASR 的 0.3–1.5s 可忽略。

### 2.2 关键设计决策

**① 本地规则 NLU，而非 LLM。** 绘图指令域封闭，规则解析 <5ms、零 API 成本、行为可预测。LLM 兜底作为规划项未实现（原因见 §7.1），`parse_clause()` 的 fallback 分支已预留统一接入点。

**② 保留模式场景图。** 每个图形是带 id 的对象（类型/坐标/大小/颜色/旋转/空心实心）。这是「把第二个方块变成黄色」「删除所有圆形」「放大一点」这类**事后修改指令**能成立的前提。

**③ 逻辑坐标系 1200×750。** 后端不知道客户端画布的物理像素，所以场景用固定逻辑坐标；前端按 `contain` 等比缩放渲染。好处：换设备/改窗口大小，同一份服务器场景都能正确还原。

**④ 可插拔持久化（部署中实际踩坑后加的）。** 服务器的源码编译版 Python 3.8 没带 `_sqlite3` C 扩展，`import sqlite3` 直接失败。于是抽象出 `store_sqlite` / `store_json` 两个后端，`store.py` 按环境自动选择：能 import sqlite3 用 SQLite，否则回退到无依赖的 JSON 文件后端。本地用 SQLite，服务器自动落到 JSON，**无需在服务器重编译 Python**。

**⑤ TTS 回声抑制。** 播报会被麦克风听到形成自我指挥的回声循环。播报开始 `recognition.abort()` 挂起识别，`onend` 后 250ms 恢复。说「关闭播报」可进入纯视觉低延迟模式。

**⑥ 「暂停聆听」是后端静音而非关麦。** 真关识别用户就无法用语音恢复了。所以识别继续运行，后端在 muted 态只响应「继续聆听」，其余指令返回空结果。

### 2.3 HTTP API

| 方法 | 路径 | 说明 |
|---|---|---|
| `GET` | `/api/healthz` | 健康检查，返回版本与当前存储后端 |
| `POST` | `/api/sessions` | 新建会话，返回 `session_id` + 初始场景 |
| `GET` | `/api/sessions/{id}` | 取会话当前场景（刷新恢复用） |
| `POST` | `/api/sessions/{id}/commands` | **核心**：提交一句语音文本，后端解析执行，返回新场景 + 每条子句结果 |
| `GET` | `/api/sessions/{id}/commands?limit=N` | 查指令审计日志 |

响应统一结构：`{session_id, scene{shapes,bg}, selected[], flags{muted,tts_on}, logical{w,h}, results[{ok,msg,intent,action,clause}]}`。

## 5. 延迟分析

| 阶段 | 耗时 | 可控性 |
|---|---|---|
| ASR（说完→final 文本） | 0.3–1.5s | ❌ 浏览器/云端决定 |
| 网络往返（前端→后端） | 局域 <5ms / 公网数十 ms | ⚠️ 取决于部署位置 |
| 后端归一化+分句+解析+执行 | **<5ms**（实测整链路 4ms） | ✅ 纯本地规则 |
| 前端渲染 | <16ms（一帧） | ✅ Canvas 2D |
| TTS 播报（可关） | 1–3s，期间挂起识别 | ✅ 说「关闭播报」 |

结论：应用自身（后端解析 + 前端渲染）引入延迟 < 一帧。整体体感延迟基本等于 ASR 本身。这正是选本地规则而非云端 LLM 的核心理由。

## 7. 未完成项与原因

**7.1 LLM 兜底解析**（规划过，主动放弃）。规则未命中时交 LLM 转结构化操作能提升长尾理解率。不做的原因：① 需 API Key 与网络，破坏零配置；② 每条多 0.5–2s 延迟，与低延迟目标冲突；③ 域内规则覆盖率已够。架构已预留 fallback 接入点。

**7.2 自由手绘曲线**。语音是离散低带宽通道，无法承载连续轨迹控制——交互模态的本质限制，非工程量问题，故判定不适合做。

**7.3 复合图案（小房子/笑脸）**。本质是模板库，每个图案需人工定义组成。为优先打磨核心编辑闭环（画→选→改→撤销→导出）而砍掉；后续以数据文件扩展即可，不动架构。

**7.4 高级样式与图层（渐变/透明度/描边宽度/深浅微调/层级/多画布）**。纯工程量：每项 = 新意图 + 新槽位 + 渲染支持。为控复杂度砍掉，场景图模型（图形即对象）已能承载这些扩展字段。

**7.5 空间指代（"左边那个圆"）**。需按坐标排序消歧，且多候选时口语本身有歧义。已有「第N个」「选中」两条等价路径，性价比不足。

**7.6 同子句内无连接词多意图**。分句依赖显式连接词（然后/接着/再/逗号）。ASR 通常会为停顿补逗号，自然语流大多能正确切分；无停顿连读属少数。完整解决需语义级切分（LLM 路线，见 7.1）。

**7.7 已知限制**
- 首次需点一次开启麦克风（浏览器安全策略，无法绕过）。
- 仅 Chrome/Edge 桌面版（Web Speech API 兼容性）；Safari 部分可用未测。
- 线上用自签名证书，首次访问需点「高级→继续前往」（生产应换正式域名 + Let's Encrypt）。
- Web Speech API 在 Chrome 上依赖云端，断网不可用（离线 ASR 见考量：40MB+ 模型、中文小模型识别率偏低，得不偿失）。

## 8. 部署架构

```
用户浏览器 ──HTTPS:8443──► 云服务器 117.50.181.92 (CentOS 7.9 / 2GB)
                            └─ systemd: voicedraw.service
                               └─ uvicorn（内置 TLS，自签名证书）
                                  └─ FastAPI app（前端静态 + API）
                                     └─ JSON 文件存储 /opt/voicedraw/data/
```

- **为什么要 HTTPS**：Web Speech API 与麦克风 `getUserMedia` 要求安全上下文（secure context），HTTP 下浏览器拒绝授权麦克风。用 uvicorn 内置 TLS + 自签名证书满足，省去 nginx（2GB 内存够用，少一个进程）。
- **为什么 JSON 而非 SQLite**：服务器源码编译的 Python 3.8 缺 `_sqlite3` 扩展，自动回退（见 §2.2④）。
- **一键部署**：`bash deploy/deploy.sh` 完成 rsync → venv(清华源) → 生成证书 → 装 systemd → 重启 → 健康检查。开机自启已 enable。
